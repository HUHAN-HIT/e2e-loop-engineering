# Craft 术语操作定义 (glossary)

> 本文件给出 Loop Engineering 各阶段反复出现、但 SKILL.md 未操作化的关键术语的**可判定定义 + 反例清单**。
> 所有 worker / coordinator 在用到这些词做"自检"时, 以本文件的判据为准, **不要凭语感判断**。
>
> 定位: 本文件是 craft 标准层 (`standards/`) 的公共底座, 与代码 SSOT (`@e2e-loop/ssot`, TS) 互补——
> TS SSOT 定义"机械检查怎么算", 本文件定义"人/agent 在写产物时怎么算做对了"。两者冲突时, 凡涉及机械求值以 TS SSOT 为准, 凡涉及 craft 判断以本文件为准。

---

## 如何读分档标记

每条判据后带 `[S]` / `[M]` / `[C]` 标记其适用复杂度档 (simple / medium / complex)。
- 只标 `[C]` 的规则: simple/medium 档**不要套用**, 否则违背"摩擦与复杂度匹配"(SKILL §5 阶段 0)。
- 标 `[S][M][C]` 的规则: 全档适用。

---

## 1. 客观可判定 (objective / mechanically-decidable)

**定义:** 一个检查项是"客观可判定"的, 当且仅当它能化约为一个**不依赖语义理解**的判断: 有/无、相等/不等、在集合内/外、数值比较、glob 是否命中。判断结果与"谁来判"无关——换一个 agent、换一次运行, 结论必然相同。

**判据 (满足全部才算客观可判定):** `[S][M][C]`
1. 输入是一个**具体字段 / 文件 / 状态值**, 不是一段自然语言。
2. 算子是固定文法之一: `存在`、`==`、`!=`、`in`、`not in`、`<`、`<=`、`>`、`>=`、`glob 命中`。
3. 右值是**字面量**或另一个具体字段, 不含函数调用、不含"是否充分/优雅/合理"这类形容。
4. 不需要读懂内容含义就能判。

**反例清单 (这些都**不是**客观可判定, 出现即退回重写):**

| ✗ 非客观 | 为什么 | ✓ 改成客观 |
| --- | --- | --- |
| "summary 是否讲清楚了" | 需语义理解 | `summary.md 存在 且 字数 <= 1200` |
| "测试是否是空壳" | 需读懂测试意图 | (交 reviewer 产 verdict, gate 只读 `verdict == "pass"`) |
| "设计是否优雅" | 偏好判断 | 不作为门禁项 (降 advisory) |
| "断言字段是否合理" | 形容词 | `assert 的字段 in 产物已声明字段集` |
| "改动范围是否过大" | 无阈值 | `actual_writes 全部命中 allowed_write_paths` |

**与 SSOT 的关系:** checks 文法的形式定义与求值见 `@e2e-loop/ssot/checklists` 的 `parseCheck` / `evalCheck`。凡写不出落入该文法的断言, 即非客观可判定。

## 2. 阻塞性歧义 (blocking ambiguity)

**定义:** 一个需求中的不确定点是"阻塞性"的, 当且仅当**不先定它, 就无法确定下游的验收口径 / 任务拆分 / 测试 / 风险判断之一**, 且没有任何"合理默认"能让开发继续而不必然返工。

**判据 (满足任一即阻塞, 否则非阻塞):** `[M][C]`
1. 答案的不同取值会**改变某条 AC 的真假口径** (验收语义分叉)。
2. 答案会**改变任务拆分结构** (多一个/少一个 task, 或依赖方向变)。
3. 答案会**改变测试该断言什么** (同一行为有互斥的正确结果)。
4. 答案会**改变风险定级** (是否触及控制面/安全/迁移/不可逆)。
且: 给不出一个"采纳它继续、错了也只是局部返工"的默认假设。

**反例清单 (这些是**非阻塞**, 应直接用合理默认继续, 不要问人):**

| ✗ 不该问 | 为什么非阻塞 | 处理 |
| --- | --- | --- |
| "按钮用什么颜色" | 不改 AC/拆分/测试/风险 | 默认 + 收口时人可改 |
| "日志用 info 还是 debug" | 可后置, 不影响验收 | 实施期自决 |
| "要不要顺便加缓存" | 是 nice-to-have, 非本需求 | 不做, 列 follow-up |
| "命名用驼峰还是下划线" | 风格, 跟随既有代码 | 跟随仓库约定 |
| "未来要不要支持多语言" | 假设性未来需求 | 不进本 run |

**经验上限:** 一次澄清 `[M]` ≤ 2 个问题, `[C]` ≤ 4 个; 超过说明在问 nice-to-have, 回头删。判据见 `clarification-standard.md`。

## 3. 关键 diff (key diff)

**定义:** 一处代码改动是"关键"的, 当且仅当它**改变了别人能观察到的契约或行为**, 或**落在高风险路径**上——即收口时人**必须**看一眼才能负责任地签字的那些改动。不是所有 diff 都关键; 把全部 diff 列进 key-diffs 等于没列。

**判据 (命中任一即为关键, 必须写进 `key-diffs.yaml`):** `[S][M][C]`
1. 改了**公共接口 / 导出符号 / API surface / 函数签名** (调用方能观察)。
2. 改了**控制流核心 / 状态机 / 调度 / 鉴权 / 校验**逻辑。
3. 改了**数据格式 / schema / 迁移 / 序列化**。
4. 落在 `risk: high` 或 `exclusive` 的 task 内 (这两类 task 的 key-diffs **必填非空且可解析**, 否则视为未提交退回——见 `@e2e-loop/ssot/checklists` 的 `validateKeyDiffsSubmission`)。
5. 引入**新依赖 / 改 lockfile / 改构建配置**。
6. 单文件改动 > ~80 行的实质逻辑 (非格式化/非生成)。

**反例清单 (这些**不是**关键 diff, 不要塞进 key-diffs 制造噪音):**

| ✗ 非关键 | ✓ 处理 |
| --- | --- |
| 改了一处内部变量名 | 不列 |
| 加了一行注释 / 调了缩进 | 不列 |
| 纯新增测试文件 | 不列 (测试结果进 test-results.yaml) |
| 等价重构、对外行为不变的内部抽取 | 一般不列; 若动了上述判据 1–5 才列 |

**每条 key-diff 的最小内容:** `{file, change, why, risk}` 四字段齐全; `change` 要能让人不打开文件就懂"改了什么"; `why` 一句话动机; `risk` ∈ {none, low, high}。

## 4. service 边界 (跨服务判定)

**定义:** 两段代码属于"不同 service", 当且仅当它们满足以下**至少一条**独立性判据。跨服务 ≥ 2 自动判 complex (SKILL §5 阶段 0)。

**判据 (满足任一即算独立 service):** `[C]`
1. **独立部署:** 各自有独立的部署单元 / 进程 / 容器, 能单独发布。
2. **契约通信:** 之间只通过明确接口 (HTTP / RPC / 消息 / 事件) 通信, 不共享内部函数调用。
3. **独立存储:** 各自拥有独立的数据存储 / schema, 不直接读写对方的表。

**反例清单 (这些**不算**多服务, 不要触发 complex 自动升档):**

| ✗ 不算多 service | 为什么 | 归类 |
| --- | --- | --- |
| monorepo 内的分层 (controller / service / dao) | 同进程内部分层, 共享调用 | **单服务**, 按 path 冲突判定 |
| 同一应用拆成多个 package / module | 编译期组织, 非独立部署 | 单服务 |
| 前端 + 后端但本 run 只动后端 | 本 run 只触及一个 service | 单服务 |
| 一个库的多个子模块 | 同发布单元 | 单服务 |

**多 repo 提示:** 真·多 service 且跨 repo 时, 加 `service → 物理树` 映射 (`@e2e-loop/ssot/multi_service` 的 `resolveWorktreeForTask`); MVP 边界建议"多服务 = monorepo", 多 repo 降级为多个独立 run。

## 5. 任务粒度合适 (right-sized task)

**定义:** 一个 task 粒度"合适", 当且仅当**一个 worker 能在隔离上下文里独立持有它的全部必读信息并一次做完**, 既不大到上下文爆掉/遗漏细节, 也不碎到徒增协调。

**判据 (合适的 task 应同时满足):** `[M][C]`
1. **单一职责:** 一句话能说清它产出什么, 不含"并且"连接的两件事。
2. **上下文可持有:** 必读切片 (context_paths + 依赖产物摘要) 加起来是一个 worker 能舒适装下的量——经验上 `[C]` task 触及文件 ≤ ~5 个主文件、改动量级 ≤ 一次 code review 能看完。
3. **写路径明确且收敛:** `allowed_write_paths` 能用具体前缀目录表达, 不靠裸 `**` (裸 `**` 会让冲突判定保守串行, 见评审 F3)。
4. **可独立验证:** 它的 AC 能在本 task 内被测试证明, 不必等别的 task 才能判对错。

**反例清单:**

| ✗ 粒度不当 | 病征 | 修法 |
| --- | --- | --- |
| "实现整个登录模块" | 太大, 一个 worker 扛不住 | 拆成 验证码生成 / 校验接口 / 登录接入 三个 task |
| "改一个变量名" | 太碎, 协调成本 > 收益 | 并进相邻 task |
| "实现 A 并顺便重构 B" | 双职责 | 拆成两个 task, 或 B 列 follow-up |
| 写路径 `src/**` | 范围失控, 必与他人冲突 | 收敛到 `src/auth/captcha/**` |
