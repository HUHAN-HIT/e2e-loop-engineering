# 澄清阶段 craft 标准 (CLARIFYING)

> 适用角色: `clarification-finder`。coordinator 判定可能存在阻塞性歧义时才分发本阶段, **多数 run 跳过**。
> 本标准回答: 什么该问、什么不该问、怎么写默认假设、问几个、产出长什么样。
> 配套: 阻塞性歧义 / 非阻塞反例的术语判据见 `glossary.md` §2。

体例: 每节给 **判据 (可自检清单) → 正/反例 → worked example**。规则带 `[S][M][C]` 档标记。

---

## 0. 一句话定位

澄清不是"把需求问全", 是"只问那些不先定就必然返工的点"。**问一个非阻塞问题的成本 = 占用人最稀缺的注意力预算 (SKILL §7)**, 所以默认倾向是**不问、用默认继续**, 只有跨过阻塞门槛才升级给人。

---

## 1. 该不该进澄清 (coordinator 先判, finder 复核)

**进澄清的判据 (满足任一):** `[M][C]`
1. 需求里有一个词/约束有 ≥2 种互斥解读, 且不同解读会改变 AC 真假口径。
2. 缺一个关键输入 (目标用户 / 数据来源 / 边界条件), 缺了它无法定测试该断言什么。
3. 需求与现状/既有契约可能冲突, 冲突解法会改变拆分。

**不进澄清 (直接默认继续):** `[S][M][C]`
- simple 档**一律跳过**澄清 (单一改动、无状态机、单服务)。
- 所有不确定点都能给出"采纳默认、错了只局部返工"的合理假设。

---

## 2. 单个问题的质量栏

**一个合格的澄清问题必须 (缺一即退回重写):** `[M][C]`
1. `why_blocking` 写清**影响哪条 AC / 哪个拆分 / 哪个测试 / 哪个风险**——不能是"想确认一下"。
2. 带一个 `default_if_unanswered`: **可直接采纳、能让开发立刻继续**的具体假设 (不是"看你"/"待定")。
3. 是**封闭或半封闭**问题 (有限选项 / 具体取值), 不是开放式"你希望它怎样"。
4. 不自答开放设计选择 (那是计划阶段的事), 只标"必须先定才能动"的点。

**好/坏 `default_if_unanswered` 对照:**

| ✗ 坏默认 | 病征 | ✓ 好默认 |
| --- | --- | --- |
| "由实现决定" | 没给假设, 等于没默认 | "默认仅校验长度 4–6 位数字验证码" |
| "待用户确认" | 把球又踢回人 | "默认验证码 5 分钟过期" |
| "尽量通用" | 不可执行 | "默认只支持邮箱登录, 手机号列 follow-up" |
| "随便" | 无信息 | "默认大小写不敏感比对" |

---

## 3. 问题数量

**经验上限 (超了说明在问 nice-to-have, 删到达标):** 
- `[M]` ≤ 2 个阻塞问题。
- `[C]` ≤ 4 个阻塞问题。
- 任何档: 若所有问题都能配好默认 → 设 `can_proceed_with_defaults: true`, 让人**一键跳过**全部回答。

**删问题的自检:** 逐条问"把这个问题的默认直接采纳, 最坏后果是什么?"——若最坏只是局部返工/收口微调, 删掉它。

---

## 4. 产出契约

`clarification/questions.json` (schema 以 `loop_engineering/schema/clarification.py` 为参考):

```json
{
  "schema": "loop-engineering.clarification.v2",
  "questions": [
    { "id": "Q1",
      "question": "验证码是纯数字还是字母数字混合?",
      "why_blocking": "影响 AC-002 的校验口径与 T01 生成逻辑的字符集",
      "default_if_unanswered": "默认 5 位纯数字" }
  ],
  "can_proceed_with_defaults": true
}
```

无阻塞歧义时返回:
```json
{ "schema": "loop-engineering.clarification.v2", "questions": [], "can_proceed_with_defaults": true }
```

---

## 5. 红线 (本阶段特有, §0 公共约定之上的追加)

- 不为"问得全"而问; 不问偏好性、可后置、假设性未来的问题 (反例见 `glossary.md` §2)。
- 不替人做开放设计决策。
- 不把一个问题拆成三个近义问句凑数。

---

## 6. Worked example

**需求 (medium):** "给登录加图形验证码。"

**finder 推理:**
- "图形验证码" → 是后端生成图片, 还是接第三方 (reCAPTCHA)? → **改拆分 + 改测试 + 改依赖**, 给不出无损默认 → **阻塞**, 问 Q1, 默认"后端自gener成 SVG, 不接第三方"。
- 验证码位数/字符集? → 改 AC-002 校验口径 → 阻塞, 问 Q2, 默认"5 位纯数字"。
- 按钮放左还是右? → 不改 AC/拆分/测试/风险 → **非阻塞**, 不问。
- 过期时间? → 能给无损默认 (5 分钟) → 不问, 写进默认随计划走。

**产出:** 2 个问题 (达标 ≤2), 都配好默认, `can_proceed_with_defaults: true`。人若不答, coordinator 采纳两个默认直接进 PLANNING。
